# 运维排班系统设计说明（需求文档，不含代码）

> 本文档用于指导开发实现一个「运维内部排班系统」。  
> 目标：不限定技术栈，仅从业务和数据设计层面说明，使任何开发同事都能据此落地实现。

---

## 1. 背景与目标

### 1.1 背景

- 运维团队需要对日常值班（含白班/夜班/周末/节假日）进行统一管理。
- 当前排班方式（例如手工 Excel / 聊天群通知）在以下方面存在不足：
  - 难以保证规则合规（加班、连续夜班等）；
  - 无法快速查看某人长期班次负荷；
  - 审批流程不透明，历史记录追溯困难。

### 1.2 目标

构建一个「运维排班系统」的基础能力，满足：

1. **排班管理**
   - 支持按月或按季度定义排班。
   - 支持白班、夜班等不同班次，支持主值班 / 副值班角色。

2. **规则约束**
   - 支持对个人每周总班次数、夜班次数、连续工作天数等做限制。
   - 支持节假日、特殊人员（如不排夜班）的例外规则。

3. **校验与审批**
   - 对排班结果进行自动校验，防止违反约束。
   - 通过版本管理（如 Git）+ 审批流程管控排班变更。

4. **展示与通知**
   - 提供按月、按人、当日等视图。
   - 提供每日自动通知当日值班人到 IM 群或门户页面（后续迭代实现）。

> 本文档不设计具体实现语言，仅规定数据模型、规则、流程和接口期望。

## 2. 排班需求与业务规则

### 2.1 覆盖范围

- **排班时间**：
  - 默认以「自然日」为单位，每天可以包含 0~N 个班次。
  - 支持两种常见模式（可根据团队实际调整）：
    - 工作日 09:00–18:00 的白班；
    - 7×24 值班（白班 + 夜班）。

- **班次类型示例**（可配置）：
  - `day`：白班（如 09:00–18:00）
  - `night`：夜班（如 18:00–09:00）
  - （可选）`weekend`：周末班（如白班单独标识）

- **角色类型示例**：
  - `primary`：主值班（Primary Oncall）
  - `secondary`：副值班（Secondary / Backup）
  - （可选）`manager_on_duty`：值班经理

### 2.2 人员与角色

- **运维成员（Ops Member）**：
  - 每个成员具有唯一 ID（如 `ops_alice`）。
  - 成员属于某个团队或职能组（SRE、DBA、Network 等）。
  - 成员可���置参与或不参与排班。

- **角色分配**：
  - 每个班次下可以配置多个角色：
    - 如 `day.primary`、`day.secondary`、`night.primary` 等。
  - 不同角色可以有不同的规则（例如 manager 班次频率低一些）。

### 2.3 约束规则（需实现校验与生成时遵守）

以下为示例规则，具体数值可在配置中定义：

1. **班次上限**
   - 每人每周最多 `max_shifts_per_week` 个班（包含白班+夜班）。
   - 每人每周最多 `max_night_shifts_per_week` 个夜班。

2. **连续工作限制**
   - 每人连续工作天数不得超过 `max_consecutive_work_days`。
   - 夜班结束后的第二天可以视为「工作日」或「休息日」，由规则配置中明确。

3. **夜班连续限制**
   - 若开启 `avoid_consecutive_nights = true`：
     - 同一成员不得连续两天担任夜班主值班（`night.primary`）。

4. **公平性规则**
   - 在一个排班周期（如一个月或一个季度）内：
     - 每人被分配的总班次数差值须在一定范围内，例如：
       - `max(total_shifts) - min(total_shifts) <= fairness_threshold`
   - 周末/节假日班次也需要尽量平均（可作为软约束，给出提示而不强制失败）。

5. **特殊日期和例外**
   - 节假日可以配置为单独的规则：
     - 只从「自愿值班名单」中轮转；
     - 或按特殊补偿规则记录（如需要导出给财务）。
   - 支持为某些成员配置「不排夜班」或其他特殊约束。

> 所有规则应支持以配置形式存在（如 JSON/YAML/数据库），而不是硬编码在程序逻辑中。

## 3. 数据模型设计

以下数据模型既可以映射为数据库表结构，也可以用作 JSON/YAML 定义。  
实际实现时，可按技术栈选择持久化方案。

### 3.1 成员模型（Member）

**字段示例：**

- `id`：字符串，唯一标识（例：`ops_alice`）
- `name`：成员姓名（例：`Alice Zhang`）
- `team`：所属团队（如 `SRE`, `DBA`, `Network`）
- `skills`：技能标签数组（例：`["db","k8s"]`）
- `is_active`：是否参与排班（布尔值）
- 可选个性化规则：
  - `max_shifts_per_week`：若为空，使用全局规则；
  - `max_night_shifts_per_week`：若为空，使用全局规则；
  - `no_nights`：布尔值，`true` 表示不排夜班。

### 3.2 规则模型（Rules）

#### 3.2.1 全局规则

- `max_shifts_per_week`：整数。
- `max_night_shifts_per_week`：整数。
- `avoid_consecutive_nights`：布尔值。
- `max_consecutive_work_days`：整数。
- `fairness_threshold`：允许的排班差异阈值。

#### 3.2.2 特殊规则

- `holiday_policy`：
  - `use_volunteers_only`：布尔值。
  - `volunteer_member_ids`：成员 ID 列表。
- `blackout_dates`：
  - 列表，每个元素：
    - `date`：日期
    - `disallow_nights`：是否禁止夜班
    - `disallow_members`：不允许参与排班的成员列表（如培训或年会期间）。

### 3.3 排班周期模型（Schedule Period）

用于标识一个排班配置的范围，例如“2026 年 4 月”。

- `id`：唯一 ID，如 `2026-04` 或 `2026-Q2`
- `type`：`month` 或 `quarter` 或其他
- `start_date`：开始日期
- `end_date`：结束日期
- `timezone`：时区（如 `Asia/Shanghai`）

### 3.4 排班结果模型（Schedule Item）

建议以「日期 + 班次 + 角色」为结构。  
一个建议的抽象结构如下：

- 顶层：
  - `period`：关联的排班周期 ID
  - `timezone`：时区
  - `items`：数组，每个元素代表��天的排班

- `item`（单日）：
  - `date`：`YYYY-MM-DD`
  - `weekday`：例如 `Mon` / `Tue` / ...
  - `shifts`：对象，键为班次类型（如 `day`, `night`）
    - 每个班次下有若干角色：
      - 例如：
        - `day.primary`: 成员 ID
        - `day.secondary`: 成员 ID
        - `night.primary`: 成员 ID
        - `night.secondary`: 成员 ID

> 实际落地时，可对应为数据库中的一张排班表，或者一个 JSON/YAML 文件。

## 4. 排班生成流程设计（逻辑描述）

本节描述自动排班生成的核心逻辑，开发需要按此实现，但不限定语言和技术栈。

### 4.1 输入

- 排班周期（开始日期 / 结束日期 / timezone）。
- 有效运维成员列表（`is_active = true`）。
- 全局规则与特殊规则配置。
- 已锁定的排班项（人工预设的安排，如某人提前申请某日值班）。

### 4.2 输出

- 一份完整的排班结果：
  - 覆盖周期内的每一天；
  - 对于每一天的每个班次和角色，给出具体成员 ID；
  - 满足或尽可能接近所有配置规则。

### 4.3 核心算法步骤（伪代码形态）

1. **初始化**
   - 从周期中生成日期列表。
   - 按周划分日期（便于每周统计）。
   - 为每个成员初始化统计信息：
     - `weekly_shifts[week_index]`
     - `weekly_night_shifts[week_index]`
     - `current_consecutive_work_days`
   - 设置初始轮转顺序（如按成员 ID 排序）。

2. **遍历日期**
   - 对每个 `date`（从 `start_date` 到 `end_date`）：
     - 计算该日期属于第几周（周索引）。
     - 初始化当天的 `shifts` 结构（`day`, `night` 等）。

3. **处理手工锁定排班（若有）**
   - 对于已手动指定的排班项：
     - 将相关成员直接填入对应班次和角色；
     - 同时更新该成员当周统计信息；
     - 若违反规则，记录为「预存在的违反项」，并在结果中附上告警。

4. **为每个班次和角色分配成员**
   - 按既定顺序或优先级（比如先白班后夜班）逐个处理：
     - 例：处理 `day.primary` -> `day.secondary` -> `night.primary` -> `night.secondary`。
   - 对于一个待分配的角色：
     1. 生成候选人列表：
        - `is_active = true`；
        - 不在该日的「不可排班」名单中；
        - 未超过当周总班次上限；
        - 若是夜班：
          - 未超过当周夜班上限；
          - 若配置禁止连续夜班，则前一日该成员未担任夜班主值班；
        - 若该角色有技能/团队要求，则满足条件（如需要至少一个具备 `db` 技能的人）。
     2. 从候选人中选择：
        - 优先排当前周期内总班次更少的成员（保证公平）；
        - 若有多个候选人总班相同：
          - 可采用轮转顺序（上一次选了 A，下次优先 B）；
          - 或根据预设优先级（如资历、意愿）。
     3. 如无候选人可选：
        - 标记该班次为「分配失败」，记录无法分配原因。
        - 算法可以继续尝试填其他班次，但最终结果��要标出「不完整」。

5. **更新统计信息**
   - 对于成功被分配的成员：
     - `weekly_shifts[week_index] += 1`；
     - 若是夜班，则 `weekly_night_shifts[week_index] += 1`；
     - 若该日出任任何角色，则增加 `current_consecutive_work_days`；
   - 若成员在某日完全未被排班：
     - 可将该成员的 `current_consecutive_work_days` 清零（视规则定义）。

6. **公平性检验与调整（可选）**
   - 生成初版排班后，统计每人总班次；
   - 若 `max(total_shifts) - min(total_shifts) > fairness_threshold`：
     - 可尝试在不违反其他规则的前提下：
       - 将个别班次从高负荷成员转移至低负荷成员；
     - 若仍无法满足，则在结果中记录「公平性告警」。

## 5. 排班校验流程设计（Validate）

无论排班是自动生成还是人工编辑，均需通过统一的校验流程。

### 5.1 校验输入

- 一份完整排班结果（排班周期、每日排班明细）。
- 全局规则配置与特殊规则配置。
- 成员列表（用于验证 ID 合法性）。

### 5.2 校验项

1. **结构校验**
   - 排班周期内的每一天都存在记录（日期连续）。
   - 每天的必需班次（如 `day`, `night`）是否有相应角色填充。
   - 每个引用的成员 ID 都存在于成员列表中。

2. **规则校验**
   - 对每人每周：
     - `总班次数 <= max_shifts_per_week`；
     - `夜班次数 <= max_night_shifts_per_week`。
   - 连续夜班：
     - 若开启 `avoid_consecutive_nights`：
       - 对每人，检查是否有连续两天担任 `night.primary`。
   - 连续工作天数：
     - 若 `current_consecutive_work_days` 大于 `max_consecutive_work_days`，则记录违规日期区间。

3. **公平性校验**
   - 统计每人总班次：
     - 若 `max(total_shifts) - min(total_shifts) > fairness_threshold`，则记录对应成员名单。

4. **特殊规则校验**
   - 节假日班次是否遵循 `holiday_policy`：
     - 若设为仅志愿者，则排班成员都应在 `volunteer_member_ids` 列表中。
   - `blackout_dates` 中标记为禁止的行为是否被触发。

### 5.3 校验输出

- 校验结果应包含：
  - `is_valid`：布尔值，表示是否通过校验；
  - `errors`：错误列表（违反硬性规则的项）；
  - `warnings`：告警列表（如公平性略有偏差、软约束未完全满足）。

- 每一条错误/告警应包含：
  - `type`：错误类型（如 `WEEKLY_LIMIT_EXCEEDED`, `CONSECUTIVE_NIGHTS`, `FAIRNESS_VIOLATION` 等）；
  - `member_id`（若相关）；
  - `date` 或日期范围；
  - `message`：人类可读说明。

## 6. 版本管理与审批流程设计

建议使用版本管理系统（如 Git 仓库）管理排班数据。

### 6.1 仓库结构建议（抽象）

> 实际路径和命名可根据公司规范调整。

- `members/`：成员配置（如 `members.yaml` 或数据库迁移脚本）
- `rules/`：排班规则配置（如 `rules-global.yaml`, `rules-2026.yaml`）
- `schedules/`：排班结果文件（按月/季度存放）
  - 例如：`schedules/2026-04.yaml`
- `docs/`：说明文档（包含本文档）

### 6.2 修改流程

1. **提出排班变更**
   - 增加一个新的排班周期文件或修改已有排班文件。
   - 提交变更为一个合并请求（PR/MR）。

2. **自动校验（CI）**
   - CI 流程读取变更中涉及的排班文件；
   - 运行排班校验流程（第 5 章设计）；
   - 校验未通过时禁止合并。

3. **人工审批**
   - 至少一个团队负责人 + 一个普通成员审批；
   - 审核重点：
     - 是否遵守加班、夜班及公平性规则；
     - 关键日期、节假日安排是否合理。

4. **变更生效**
   - 合并后，最新排班即视为「正式排班」；
   - 若有下游系统（告警平台、IM 通知），可以由后续流程同步数据。

## 7. 展示与通知设计

本节描述期望实现的对外功能，供前端或平台系统集成参考。

### 7.1 展示视图需求

至少需要以下三种视图：

1. **按月视图（日历视图）**
   - X 轴：日期（1~31）
   - Y 轴：每个日期显示各班次与主/副值班人。
   - 支持按团队过滤（只看 SRE、只看 DBA 等）。

2. **按人视图（个人排班表）**
   - 输入：成员 ID + 时间范围（如 2026 年 4 月）。
   - 输出：该成员所有班次列表（日期、班次类型、角色）。

3. **当日视图（Today）**
   - 默认返回今天的所有值班信息：
     - 今���主值班/副值班；
     - 夜班交接信息（如前一晚是谁，下一晚是谁）。

### 7.2 API 设计示例（仅接口形态）

具体协议可以为 HTTP/JSON，也可以是内部服务接口。

1. `GET /schedule?period=2026-04`
   - 返回指定周期的完整排班。

2. `GET /schedule/member/{member_id}?period=2026-04`
   - 返回指定成员在周期内的排班列表。

3. `GET /schedule/today`
   - 返回当日所有运维值班信息。

### 7.3 通知机制（后续迭代）

- **定时通知**
  - 每天早上指定时间（如 09:00）：
    - 读取当日排班信息；
    - 格式化为一段文本；
    - 发送到：
      - IM 群（企业微信/钉钉/飞书/Slack 等）；
      - 或内部 Portal 公告区。

- **触发方式**
  - 使用企业内部的定时任务平台；
  - 或使用 CI/Job 工具（如 Jenkins、GitHub Actions 等）定时拉取排班数据并调用通知接口。

- **消息内容示例**
  - 日期 + 星期；
  - 班次信息（白班/夜班）；
  - 每个班次的主/副值班成员；
  - 紧急联系方式或链接（如值班手册地址）。

## 8. MVP 范围与迭代计划

### 8.1 最小可行版本（MVP）

在第一个版本中，建议仅实现以下内容：

1. **手工排班 + 校验**
   - 排班数据由人工编辑（如 YAML/Excel -> 导入）；
   - 实现完整的校验逻辑（第 5 章）；
   - 确保所有硬性规则得以执行。

2. **版本管理与审批**
   - 排班变更全部通过版本管理系统操作；
   - 引入自动校验 + 人工审批流程。

3. **简单展示**
   - 至少提供一个按人视图或按月的简单 Web 页面 / 报表。

4. **手动通知**
   - 值班负责人每天手动从系统查询当日值班信息，并贴到 IM 群。

### 8.2 第二阶段迭代

在 MVP 稳定后，可以考虑：

1. 引入「自动排班生成算法」，根据规则自动计算初版排班。
2. 增强前端展示（支持日历拖拽调整、批量编辑等）。
3. 将通知机制自动化（定时推送每日值班信息）。

## 9. 非功能性需求（可选）

- 权限控制：
  - 普通成员只读排班信息；
  - 组长/管理员可编辑排班和规则。
- 日志与审计：
  - 所有排班变更需记录变更人、时间和变更内容。
- 性能与规模：
  - 支持上百名成员、跨多团队的排班；
  - 生成/校验过程在可接受时间内完成（如 1 分钟以内）。
